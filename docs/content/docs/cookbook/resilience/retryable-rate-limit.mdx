Use retryable rate limit when an external API returns 429 and you want the runtime to automatically reschedule the step after the specified delay.

## Pattern

Keep the workflow itself boring. The CRM step can throw `RetryableError` on 429, and the workflow code stays a normal fetch-then-store sequence.

```typescript lineNumbers
type Contact = { id: string; email: string };

declare function fetchContactFromCrm(contactId: string): Promise<Contact>; // @setup throws RetryableError on 429
declare function upsertIntoWarehouse(contact: Contact): Promise<void>; // @setup

export async function syncCrmContact(contactId: string) {
  "use workflow";

  const contact = await fetchContactFromCrm(contactId);
  await upsertIntoWarehouse(contact);

  return { contactId, status: "synced" };
}
```

## Key APIs

- [`"use workflow"`](/docs/api-reference/workflow/use-workflow) — declares the orchestrator function
- [`RetryableError`](/docs/api-reference/workflow/retryable-error) — lets a step ask the runtime to retry after a delay
