A workflow that sends an email and waits 3 days for a response. In a traditional system, a worker process sits idle for 72 hours. In Workflow DevKit, nothing exists during those 72 hours.

## The Key Insight

Wall-clock time is free because compute exists only at decision points.

A workflow has two kinds of moments: **decision points** — when it dispatches steps, collects results, evaluates conditions, or advances to the next operation — and **gaps** — the time between those moments where nothing happens. Traditional orchestrators keep a process alive during the gaps. Workflow DevKit doesn't.

When the workflow hits a step or `sleep()`, it records the pending operation as an event, returns `timeoutSeconds` to the queue, and exits. The handler is done. The compute is freed. No process sleeps, no connection is held, no memory is allocated.

```ts
// The handler returns timeoutSeconds — then compute goes to zero
const minTimeoutSeconds = waitItems.reduce<number | null>(
  (min, queueItem) => {
    const resumeAtMs = queueItem.resumeAt.getTime();
    const delayMs = Math.max(1000, resumeAtMs - now);
    const timeoutSeconds = Math.ceil(delayMs / 1000);
    if (min === null) return timeoutSeconds;
    return Math.min(min, timeoutSeconds);
  },
  null
);
```

When a step completes, the step handler re-enqueues the workflow. A fresh invocation loads the event log, replays through cached results in milliseconds, and resumes at the next decision point. For timed waits, the queue delivers the next message after the delay — no polling, no sleeping process.

## Why It Matters

A `sleep('7d')` costs the same as `sleep('5s')` in compute — both produce a delayed queue message and release all resources. Your workflow cost is proportional to the work performed at decision points, not the wall-clock time between them. No always-on workers. No idle connections. No paying to wait.
