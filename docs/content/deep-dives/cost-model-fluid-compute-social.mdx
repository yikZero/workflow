A workflow that sends an email and waits 3 days for a response. In a traditional system, a worker sits idle for 72 hours. In Workflow DevKit, nothing exists during those 72 hours.

## The Key Insight

Wall-clock time is free because compute exists only at decision points.

A workflow has **decision points** ‚Äî dispatching steps, collecting results, advancing ‚Äî and **gaps** between them. Traditional orchestrators keep a process alive during gaps. Workflow DevKit doesn't.

Step? The handler persists `step_created`, runs it out-of-band, then wakes the workflow when `step_completed` lands and the step handler re-enqueues the run:

```ts
// From packages/core/src/runtime/step-handler.ts
if (EntityConflictError.is(err)) {
  runtimeLogger.debug(
    'Step in terminal state, re-enqueuing workflow',
    {
      stepName,
      stepId,
      workflowRunId,
      error: err.message,
    }
  );

  await queueMessage(world, getWorkflowQueueName(workflowName), {
    runId: workflowRunId,
    traceCarrier: await serializeTraceCarrier(),
    requestedAt: new Date(),
  });

  return;
}
```

Sleep? The handler persists `wait_created`, returns `timeoutSeconds`, and lets the queue deliver the next message later. Either way, the compute is freed. No process sleeps, no connection is held. Rare edge case: hook conflicts force an immediate replay with `timeoutSeconds: 0`.

## The Three Mechanics

Three runtime paths create the cost model:

1. **Queue-driven invocation** ‚Äî `start()` persists `run_created`, queues the workflow, and returns immediately.

```ts
// From packages/core/src/runtime/start.ts
const result = await world.events.create(
  runId,
  {
    eventType: 'run_created',
    specVersion,
    eventData: {
      deploymentId: deploymentId,
      workflowName: workflowName,
      input: workflowArguments,
      executionContext: { traceCarrier, workflowCoreVersion },
    },
  },
  { v1Compat }
);

await world.queue(
  getWorkflowQueueName(workflowName),
  {
    runId,
    traceCarrier,
  } satisfies WorkflowInvokePayload,
  {
    deploymentId,
  }
);

return new Run<TResult>(runId);
```

2. **Suspend on external work** ‚Äî a step persists `step_created`, queues step execution, and exits. When orchestration hits a hook, the runtime records hook state durably and exits. The workflow stays suspended until external delivery records `hook_received`.
3. **Delay only for waits** ‚Äî `sleep()` persists `wait_created`, returns `timeoutSeconds`, and lets the queue schedule the next wake-up.

The important nuance: steps, hooks, and waits all suspend durably, but they do not resume through the same path. Steps wake the workflow via `step_completed`. Waits wake the workflow via `timeoutSeconds`.

```ts
// From packages/core/src/runtime/suspension-handler.ts
export interface SuspensionHandlerResult {
  timeoutSeconds?: number;
}

const now = Date.now();
const minTimeoutSeconds = waitItems.reduce<number | null>(
  (min, queueItem) => {
    const resumeAtMs = queueItem.resumeAt.getTime();
    const delayMs = Math.max(1000, resumeAtMs - now);
    const timeoutSeconds = Math.ceil(delayMs / 1000);
    if (min === null) return timeoutSeconds;
    return Math.min(min, timeoutSeconds);
  },
  null
);

if (hasHookConflict) {
  return { timeoutSeconds: 0 };
}

if (minTimeoutSeconds !== null) {
  return { timeoutSeconds: minTimeoutSeconds };
}

return {};
```

<Callout type="info">
Hooks are durable suspension points, not queued jobs. The create phase records `hook_created`; the receive phase records `hook_received`. Unlike a step, hook creation does not itself queue executable work.
</Callout>

## Why It Matters

A `sleep('7d')` costs the same as `sleep('5s')` ‚Äî both produce a delayed queue message and release all resources. On Vercel, waits longer than 23 hours are chained across multiple delayed messages:

```ts
// From packages/world-vercel/src/queue.ts
const MAX_DELAY_SECONDS = Number(
  process.env.VERCEL_QUEUE_MAX_DELAY_SECONDS || 82800
);

if (typeof result?.timeoutSeconds === 'number') {
  const delaySeconds =
    result.timeoutSeconds > 0
      ? Math.min(result.timeoutSeconds, MAX_DELAY_SECONDS)
      : undefined;

  await queue(queueName, payload, { deploymentId, delaySeconds });
}
```

Replay through cached step results takes milliseconds. This design can materially reduce cost versus always-on workers because compute is billed at decision points, not while the workflow is waiting. No always-on worker polling. No sleeping process. No paying to wait.

A week-long workflow. Milliseconds of compute. That's the model.
