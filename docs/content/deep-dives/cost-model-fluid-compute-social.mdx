A workflow that sends an email and waits 3 days for a response. In a traditional system, a worker process sits idle for 72 hours. In Workflow DevKit, nothing exists during those 72 hours.

## How It Works

Workflow handlers are invoked by queue messages, not long-lived processes. When the workflow hits a step or a `sleep()`, it records the pending operation as an event, returns `timeoutSeconds` to the queue, and exits. The compute is freed immediately.

```ts
// From packages/core/src/runtime/suspension-handler.ts
const minTimeoutSeconds = waitItems.reduce<number | null>(
  (min, queueItem) => {
    const resumeAtMs = queueItem.resumeAt.getTime();
    const delayMs = Math.max(1000, resumeAtMs - now);
    const timeoutSeconds = Math.ceil(delayMs / 1000);
    if (min === null) return timeoutSeconds;
    return Math.min(min, timeoutSeconds);
  },
  null
);
```

When a step completes, the step handler re-enqueues the workflow. A fresh invocation loads the event log, replays through all completed steps (returning cached results in milliseconds), and continues execution. For timed waits, the queue delivers the next message after the delay — no polling, no sleeping process.

Sleeps longer than 23 hours chain automatically. Each delayed message fires, checks if `now >= resumeAt`, and either continues or schedules another delay. A `sleep('30d')` becomes ~32 brief invocations spread across a month.

## Why It Matters

Wall-clock time is free. A `sleep('7d')` costs the same as `sleep('5s')` in compute — both produce a delayed queue message and release all resources. Your workflow cost is proportional to the work performed, not the time elapsed. No always-on workers. No idle connections. No paying to wait.
