A workflow that sends an email and waits 3 days for a response. In a traditional system, a worker sits idle for 72 hours. In Workflow DevKit, nothing exists during those 72 hours.

## The Key Insight

Wall-clock time is free because compute exists only at decision points.

A workflow has **decision points** — dispatching steps, collecting results, advancing — and **gaps** between them. Traditional orchestrators keep a process alive during gaps. Workflow DevKit doesn't.

Step? The handler persists `step_created`, runs it out-of-band, then wakes the workflow when `step_completed` lands and the step handler re-enqueues the run. Sleep? The handler persists `wait_created`, returns `timeoutSeconds`, and lets the queue deliver the next message later. Either way, the compute is freed. No process sleeps, no connection is held. Rare edge case: hook conflicts force an immediate replay with `timeoutSeconds: 0`.

## The Three Mechanics

Three runtime properties compose into the cost model:

1. **Queue-driven invocation** — workflow handlers are invoked by queue messages, not by long-lived processes. No message, no compute.
2. **Zero-cost suspension** — when a workflow encounters a step or hook, it records the pending operations as events, queues the work out-of-band, and exits. When a step completes, the step handler writes a `step_completed` event and re-enqueues the workflow — a fresh invocation picks up where it left off. No thread or connection is held between decision points.
3. **Delayed re-enqueue** — for timed waits like `sleep()`, the handler returns a `timeoutSeconds` delay and the queue delivers the next message after that interval. The workflow process does not exist during the gap.

```ts
// Only timed waits produce a timeoutSeconds — step suspension returns {}
const minTimeoutSeconds = waitItems.reduce<number | null>(
  (min, queueItem) => {
    const resumeAtMs = queueItem.resumeAt.getTime();
    const delayMs = Math.max(1000, resumeAtMs - now);
    const timeoutSeconds = Math.ceil(delayMs / 1000);
    if (min === null) return timeoutSeconds;
    return Math.min(min, timeoutSeconds);
  },
  null
);
```

Step suspension and timed waits follow different re-entry paths. When a step completes, the step handler writes a `step_completed` event and re-enqueues the workflow — no timeout involved. For timed waits like `sleep()`, the handler returns `timeoutSeconds` and the queue delivers the next message after the delay. Either way, a fresh invocation loads the event log, replays through cached results in milliseconds, and resumes at the next decision point. No polling, no sleeping process.

## Why It Matters

A `sleep('7d')` costs the same as `sleep('5s')` — both produce a delayed queue message and release all resources. Replay through cached step results takes milliseconds. Your cost is proportional to work performed, not wall-clock time. No always-on workers. No paying to wait.

A week-long workflow. Milliseconds of compute. That's the model.
